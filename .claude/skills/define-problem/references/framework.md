# 문제 정의 프레임워크 (Problem Definition Framework)

## 핵심 원리

**문제 = 간극(Gap)**

모든 문제의 본질은 "현재 상태"와 "바람직한 상태" 사이의 간극이다. 간극이 없으면 문제가 아니다. 간극을 구체적으로 말할 수 없으면 아직 문제를 이해하지 못한 것이다.

---

## 구조

### 1. 현상 (Observation)

> 지금 무슨 일이 일어나고 있는가?

객관적 사실만 기술한다. 해석, 판단, 감정을 섞지 않는다.

**좋은 예:**
- "팀 회의가 매주 평균 2시간 15분 진행되고 있다"
- "지난 분기 고객 문의 응답 시간이 평균 48시간이다"
- "신규 입사자가 3개월 내 퇴사하는 비율이 40%이다"

**나쁜 예:**
- "회의가 비효율적이다" — 판단이 섞여 있다
- "고객 서비스가 형편없다" — 감정이 섞여 있다
- "온보딩이 실패하고 있다" — 해석이 섞여 있다

**검증 질문:** 이 문장에서 누구나 동의할 수 있는 측정 가능한 사실만 남아 있는가?

---

### 2. 기대 (Expectation)

> 어떠해야 하는가?

현상과 대비되는 바람직한 상태. 이 기대가 어디서 오는지(목표, 기준, 합의, 상식)도 함께 밝힌다.

**좋은 예:**
- "회의는 40분 안에 안건이 정리되어야 한다 (팀 합의 기준)"
- "고객 문의에 24시간 내 첫 응답 (서비스 수준 협약)"
- "신규 입사자 3개월 내 퇴사율 10% 이하 (업계 평균 기준)"

**기대의 검증:**
- 이 기대는 어디서 왔는가? (합의, 데이터, 관행, 직관)
- 직관에서 왔다면 — 데이터나 합의로 뒷받침할 수 있는가?
- 이 기대를 달성한 사례가 있는가? (자기 자신 또는 비교 대상)
- 관련된 사람들이 이 기대에 동의하는가?

기대의 근거가 약하면, 간극 분석 전에 기대부터 재설정한다.

**검증 질문:** 이 기대의 근거는 무엇인가? 합리적이고 달성 가능한가?

---

### 3. 간극 (Gap)

> 현상과 기대 사이의 차이는 무엇인가?

이것이 **진짜 문제의 정체**다. 간극을 구체적으로 서술할 수 있어야 원인 분석과 해결이 의미를 가진다.

**정량적 간극** — 수치로 표현 가능:
- "회의 시간이 기대 대비 매주 95분 초과하고 있다"
- "응답 시간이 기준 대비 24시간 초과하고 있다"
- "퇴사율이 목표 대비 30%p 높다"

**정성적 간극** — 수치화 어려움. 상태를 구체적으로 서술한다:
- "의사결정 시 누가 최종 결정권을 가지는지 합의가 없어, 같은 논의가 반복된다"
- "신규 입사자가 첫 업무를 시작하기까지 누구에게 무엇을 물어야 하는지 모른다"

정성적 간극의 검증: 관련된 사람 3명에게 같은 간극을 설명했을 때, 동의하는가?

**검증 질문:** 이 간극의 크기를 수치 또는 구체적 상태 서술로 표현할 수 있는가?

---

### 4. 영향 (Impact)

> 이 간극이 왜 중요한가?

간극을 방치하면 어떤 결과가 생기는가. 누가 어떻게 영향을 받는가. 이 단계가 문제 해결의 **우선순위**를 결정한다. 영향이 작으면 해결할 가치가 없을 수도 있다.

**구성 요소:**
- **범위:** 누가, 얼마나 많은 사람이 영향을 받는가
- **심각도:** 영향의 정도는 어떠한가
- **긴급도:** 시간이 지나면 악화되는가, 유지되는가
- **비가역성:** 되돌릴 수 없는 결과를 초래하는가

**예시:**
- **범위:** 팀원 5명 전원
- **심각도:** 월 24시간의 중복 작업, 마감 평균 3일 지연
- **긴급도:** 팀 규모가 커지면서 악화 추세
- **비가역성:** 반복되면 팀원 간 신뢰 손실로 협업 의지 저하

**검증 질문:** 이 문제를 해결하지 않으면 6개월 후에 어떻게 되는가?

---

### 5. 원인 (Cause)

> 이 간극은 왜 존재하는가?

증상이 아닌 근본 원인을 찾는다. "왜?"를 반복해서 파고든다.

**증상 vs 원인 구분:**

| 증상 | 근본 원인 |
|------|-----------|
| 회의가 길다 | 안건 없이 회의를 시작한다 |
| 매출이 떨어졌다 | 주력 제품의 시장 점유율이 신규 경쟁자에게 이동했다 |
| 직원이 자주 그만둔다 | 성장 경로가 불명확하여 6개월 시점에 정체감을 느낀다 |

**원인 분석 도구:**
- **5 Whys:** "왜?"를 5번 반복하여 표면 아래를 판다
- **뼈대 분석(Fishbone):** 사람, 프로세스, 환경, 도구 등 범주별로 원인을 분류한다

**원인이 여러 개일 때:**

1. 원인을 나열한다
2. 각 원인의 기여도를 판단한다 (높음/중간/낮음)
3. 각 원인의 제거 가능성을 판단한다 (높음/중간/낮음)
4. 기여도 높고 제거 가능한 원인부터 공략한다

| 원인 | 기여도 | 제거 가능성 | 우선순위 |
|------|--------|-------------|----------|
| (원인 A) | 높음 | 높음 | 1 |
| (원인 B) | 중간 | 낮음 | 3 |
| (원인 C) | 중간 | 중간 | 2 |

**검증 질문:** 이 원인을 제거하면 간극이 실제로 줄어드는가?

---

### 6. 해결 방향 (Direction)

> 간극을 어떻게 좁힐 것인가?

원인에 대응하는 구체적 행동. 2-3개의 대안을 검토하고 선택한다.

**대안 비교 템플릿:**

| 기준 | 대안 A | 대안 B | 대안 C |
|------|--------|--------|--------|
| 원인 대응도 | | | |
| 실행 가능성 | | | |
| 비용 | | | |
| 부작용 | | | |
| 지속 가능성 | | | |

**예시 (팀 정보 공유 문제):**

| 기준 | A: 공유 보드 도입 | B: 일일 이메일 요약 | C: 전담 담당자 지정 |
|------|-------------------|---------------------|---------------------|
| 원인 대응도 | 근본 원인 제거 | 증상 완화 | 부분 대응 |
| 실행 가능성 | 지금 바로 | 지금 바로 | 채용/배정 필요 |
| 비용 | 낮음 | 낮음 | 높음 (인력) |
| 부작용 | 기록 습관 필요 | 이메일 과부하 가능 | 병목 발생 가능 |
| 지속 가능성 | 구조적 변화 | 반복 작업 | 담당자 부재 시 중단 |

**선택 원칙:**
1. 원인에 직접 대응하는 안을 우선한다
2. 비용이 같다면 단순한 안을 선택한다
3. 확신이 없으면 가장 작은 범위로 먼저 시도하고 결과를 본다

**검증 질문:** 이 해결책은 원인을 제거하는가, 아니면 증상을 완화하는가?

---

### 7. 성공 기준 (Success Criteria)

> 해결되었음을 어떻게 알 수 있는가?

간극이 줄어들었는지 판단하는 구체적이고 측정 가능한 기준.

**정량적 성공 기준:**
- "회의 시간이 평균 45분 이하로 줄었다 (4주간 측정)"
- "고객 첫 응답 시간이 24시간 이내이다 (95th percentile)"
- "3개월 내 퇴사율이 15% 이하이다 (2분기 연속)"

**정성적 성공 기준** — 수치화 어려운 간극의 경우:
- "의사결정 시 최종 결정권자가 누구인지 팀원 전원이 답할 수 있다 (개별 확인)"
- "신규 입사자가 첫 업무 착수까지 질문할 대상을 스스로 찾을 수 있다 (입사 2주 후 인터뷰)"

정성적 기준의 검증: 관련된 사람에게 직접 물어서 확인할 수 있는 형태로 작성한다.

**검증 질문:** 이 기준을 충족했는지 누구나 객관적으로 판단할 수 있는가?

## 사용 가이드

### 순서가 중요하다

```
현상 → 기대 → 간극 → (여기까지가 문제 정의)
 ↑             ↑    → 영향 → (해결할 가치가 있는가?)
 │             │           → 원인 → (왜 발생하는가?)
 │             │                  → 해결 방향 → 성공 기준
 │             │                        │
 └─────────────┴── 필요시 이전 단계로 ──┘
```

**간극이 명확해지기 전에 원인이나 해결책으로 넘어가지 않는다.** 이것이 가장 중요한 규칙이다.

### 이전 단계로 돌아가는 것은 정상이다

이 프레임워크는 선형이 아니라 나선형이다.

- 원인을 분석하다가 → 현상을 다시 관찰해야 할 수 있다
- 영향을 판단하다가 → 원인을 알아야 정확해질 수 있다
- 해결 방향을 검토하다가 → 기대를 조정해야 할 수 있다

순서는 "처음 접근할 때의 권장 순서"이지, 되돌아가면 안 된다는 뜻이 아니다. 단, **간극이 명확해지기 전에 원인/해결로 넘어가지 않는다는 규칙은 유지한다.**

### 규모에 따른 적용

| 규모 | 필수 요소 | 선택 요소 |
|------|-----------|-----------|
| 작은 문제 (개인, 일상) | 현상, 기대, 간극, 해결 방향 | 나머지 |
| 중간 문제 (팀, 프로젝트) | 전체 7단계 | — |
| 큰 문제 (조직, 사회) | 전체 7단계 + 각 단계의 세부 분석 | — |

작은 문제라도 **현상-기대-간극** 3단계는 반드시 거친다. 원인이 자명하지 않으면 작은 문제라도 5단계(원인)를 거친다.

### 이 프레임워크를 생략해도 되는 경우

- **즉각 행동이 필요한 긴급 상황:** 서버 장애, 안전 사고 등. 먼저 대응하고, 사후에 이 구조로 분석한다.
- **원인과 해결이 자명한 경우:** "오타가 있다 → 고친다." 현상-기대-간극이 한 문장으로 끝나면 나머지 단계는 불필요하다.
- **탐색 단계:** 아직 현상 자체를 파악 중이면, 1단계(현상)에 머물면서 관찰을 계속한다. 억지로 7단계를 채우지 않는다.

### 흔한 실수와 대응

| 실수 | 예시 | 대응 |
|------|------|------|
| 해결책을 문제로 위장 | "Slack이 필요하다" | "어떤 상황에서 무엇이 안 되고 있는가?" |
| 증상을 원인으로 착각 | "두통이 문제다" | "왜 두통이 생기는가?" |
| 판단을 사실로 포장 | "분위기가 안 좋다" | "무엇이 관찰되었는가?" |
| 너무 넓게 정의 | "소통이 안 된다" | "누가, 언제, 어떤 소통이 안 되는가?" |
| 너무 좁게 정의 | "이 버튼 색을 바꿔야 한다" | "사용자가 실제로 겪는 어려움은 무엇인가?" |
| 간극 없이 해결책 제시 | "주간 보고서를 만들자" | "지금 어떤 정보가 부족한가?" |

## 적용 예시

### 예시 1: 팀 내 정보 공유

**현상:** 팀원 A가 완료한 작업을 팀원 B가 모르고 같은 작업을 다시 수행한 사례가 지난 한 달간 3건 발생했다.

**기대:** 한 사람이 완료한 작업은 같은 날 안에 관련 팀원 전원이 인지해야 한다.

**간극:** 작업 완료 정보가 관련 팀원에게 전달되기까지 평균 3일이 걸리며, 전달되지 않는 경우도 있다.

**영향:** 월 약 24시간의 중복 작업 발생. 팀원 간 신뢰 저하. 마감 지연.

**원인:** 작업 완료 시 공유하는 합의된 채널이나 절차가 없다. 각자 개인 메모에 기록하고 구두로만 공유한다.

**해결 방향:** 작업 완료 시 공유 보드에 기록하는 절차를 합의하고, 일일 스탠드업에서 확인한다.

**성공 기준:** 중복 작업 발생 건수가 월 1건 이하 (3개월간 측정).

---

### 예시 2: 간극이 불명확한 경우 (기대로 되돌아가기)

**현상:** 프로젝트 마감이 최근 3건 연속 지연되었다.

**기대 (초기):** 예정된 마감일에 완료.

**간극:** 불명확. 마감이 지연된 것인가, 애초에 마감이 비현실적이었는가?

→ 간극을 확정하기 전에 **기대(2단계)로 돌아간다.** 과거 유사 프로젝트 5건의 실제 소요 기간을 확인한 결과, 평균 소요 기간이 설정된 마감보다 1.5배 길었다.

**기대 (재설정):** 과거 실적 기반으로 마감을 설정한다. 유사 프로젝트 평균 소요 기간의 1.2배를 마감으로 잡는다.

**간극 (재산정):** 현재 마감 설정 방식에 과거 실적 데이터가 반영되지 않고 있다.

**영향:** 반복적 마감 지연으로 이해관계자 신뢰 저하. 팀원의 야근 증가로 피로 누적.

**원인:** 마감일을 프로젝트 범위와 과거 데이터 없이 희망 일정으로 설정하고 있다.

**해결 방향:** 프로젝트 착수 시 유사 과거 프로젝트의 실제 소요 기간을 조회하고, 이를 기반으로 마감을 산정하는 절차를 도입한다.

**성공 기준:** 마감 준수율이 5건 중 4건 이상 (80%)으로 향상된다 (6개월간 측정).

---

### 예시 3: 원인이 복수인 경우

**현상:** 고객 이탈률이 분기 15%이다.

**기대:** 분기 5% 이하 (업계 상위 25% 기준).

**간극:** 이탈률이 목표 대비 10%p 높다.

**영향:** 연간 매출 손실 추정 약 2억 원. 신규 고객 획득 비용이 유지 비용의 5배.

**원인 분석:**

| 원인 | 기여도 | 제거 가능성 | 우선순위 |
|------|--------|-------------|----------|
| 경쟁사 가격 인하 | 높음 | 낮음 | 3 |
| 고객 문의 응답 지연 | 중간 | 높음 | 1 |
| 핵심 기능 부족 | 중간 | 중간 | 2 |
