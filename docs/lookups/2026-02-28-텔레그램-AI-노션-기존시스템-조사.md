# Telegram + AI + Notion 태스크/일정 관리 기존 시스템 조사

## 조사 목적

"텔레그램 메시지 -> AI 파싱 -> Notion 데이터베이스" 패턴을 구현한 기존 시스템, 오픈소스 프로젝트, n8n 워크플로우 템플릿, 유사 도구를 조사하여 설계 참고자료로 활용한다.

## 5. 핵심 교훈 및 시사점

### 5-1. 아키텍처 패턴

1. **가장 성숙한 패턴**: Telegram Trigger -> Switch(텍스트/음성) -> (음성이면 Whisper 전사) -> AI Agent(LLM + Memory + Tools) -> Notion Tool(CRUD) -> Telegram 응답
2. **Tool Calling이 대세**: n8n의 최신 템플릿들은 대부분 AI Agent 노드 + Tool Calling 패턴을 사용. LLM이 자연어를 이해하고 적절한 도구를 선택하여 호출.
3. **메모리 필수**: 대화 컨텍스트를 유지하지 않으면 "그거 완료해줘" 같은 참조형 명령 처리 불가.

### 5-2. 흔한 문제점

1. **n8n 커뮤니티 보고**: AI Agent가 가끔 도구를 호출하지 않고 임의 답변을 생성하는 문제
2. **날짜 파싱 오류**: "next Monday", "내일" 같은 상대 날짜를 잘못 변환하는 사례 다수
3. **다수 워크플로우 충돌**: 같은 Telegram 봇에 여러 워크플로우가 연결되면 트리거 충돌
4. **토큰 한계**: 대화가 길어지면 LLM 컨텍스트 윈도우 초과
5. **Todoist Ramble 사례**: 실세계 음성에서 end-to-end 성공률이 62%에 불과 -> 반복 개선 필수

### 5-3. 설계 권장사항

1. **Approval 패턴 적극 활용**: 중요한 액션(삭제, 수정) 전에 사용자 확인 단계 삽입
2. **Add부터 시작**: 대부분의 프로젝트가 "추가"만 지원. 이것만 잘 해도 핵심 가치 제공.
3. **2단계 확장**: Add -> Complete/Query -> Update/Delete 순으로 점진적 확장
4. **음성 지원은 높은 가치**: 음성 메모 -> 전사 -> AI 파싱은 마찰을 극적으로 줄임
5. **태그/카테고리 자동 분류**: 태스크 생성 시 카테고리 자동 감지가 유용 (#7925 사례)
